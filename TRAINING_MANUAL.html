What is Vibe-Coding?

Vibe-coding is a development methodology where a human collaborates with an AI coding assistant (Claude Code) using natural language to build real software. The human describes intent, makes decisions, and provides judgment. Claude Code reads files, writes code, runs commands, and executes tasks.

This is not copy-pasting code snippets from ChatGPT. It is a genuine engineering partnership. The human remains the product owner, architect, and quality gate. Claude Code is the engineer who never sleeps, never gets tired, and can switch between debugging a YAML file and rewriting a Gemini prompt without losing context.

The Human-AI Division of Work

Human (Felix)Claude Code
Decides WHAT to build and WHYDecides HOW to implement it
Provides domain knowledge and tasteProvides technical knowledge and execution
Approves or rejects proposalsProposes options with trade-offs
Provides credentials and accessNever stores or hardcodes secrets
Tests the final product visuallyTests programmatically (curl, node, build)
Decides when to shipEnsures code is shippable before recommending
💡
The Golden Rule of Vibe-Coding

The human never writes code. Claude Code never makes product decisions. When they stay in their lanes, progress is fast and quality is high. When they blur (Claude makes architectural decisions without asking, or the human starts editing files directly), things get confused.

The Supervision Model Used in This Project

1
Read the spec. Every Claude Code session starts by reading CLAUDE.md — the master spec that tells Claude everything about the project, its rules, its architecture, and what must not be changed.
2
Confirm before acting. For any significant change — architectural decisions, new workflows, touching production infrastructure — Claude Code confirms its plan with Felix before writing a single line of code.
3
Read before writing. Claude Code reads every file it is about to modify before touching it. This is non-negotiable — editing a file you haven't read causes corruption.
4
Test before committing. Every code change is verified (npm run build, node imports, curl tests) before a git commit. Never commit broken code.
5
Log everything verbatim. Every session is recorded in SESSION_LOG.html with every command, every output, every decision, every error, and every fix — in full. No summaries.

Building the Blog Frontend

The Supabase Lazy Proxy Pattern

A critical lesson learned during the first build attempt: if you call createClient() at module import time, Next.js will crash during next build because environment variables are not available at build time.

The naive (broken) approach:

import { createClient } from '@supabase/supabase-js'

// ❌ This runs at module load — crashes during next build
export const supabase = createClient(
  process.env.NEXT_PUBLIC_SUPABASE_URL,
  process.env.NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY
)

The correct approach — lazy initialisation using a Proxy:

lib/supabase.ts
import { createClient, SupabaseClient } from '@supabase/supabase-js'

let _client: SupabaseClient | null = null

function getClient(): SupabaseClient {
  if (!_client) {
    const url = process.env.NEXT_PUBLIC_SUPABASE_URL
    const key = process.env.NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY
    if (!url || !key) throw new Error('Missing Supabase env vars')
    _client = createClient(url, key)
  }
  return _client
}

// The Proxy intercepts every method call and delegates to the real client
// The real client is only created on FIRST USE, not at import time
export const supabase = new Proxy({} as SupabaseClient, {
  get(_target, prop) {
    const client = getClient()
    const value = (client as any)[prop]
    return typeof value === 'function' ? value.bind(client) : value
  },
})
✅
Why this works

The Proxy object is created immediately at import time (no env vars needed). The real createClient() call is deferred until the first database operation, which only happens at request time — when env vars are available.

The Protected API Endpoint

The Scout Agent publishes posts by calling a protected Next.js API route. This is how you secure an API route with an API key check:

app/api/posts/create/route.ts
export async function POST(req: Request) {
  // 1. Check API key — reject immediately if missing or wrong
  const apiKey = req.headers.get('x-api-key')
  if (!apiKey || apiKey !== process.env.SCOUT_API_KEY) {
    return NextResponse.json({ error: 'Unauthorized' }, { status: 401 })
  }

  // 2. Parse body
  const body = await req.json()

  // 3. Validate required fields
  const VALID_CATEGORIES = ['ai-models', 'anthropic', 'industry', 'tools']
  if (!body.title) return NextResponse.json({ error: 'title required' }, { status: 400 })
  if (!VALID_CATEGORIES.includes(body.category)) {
    return NextResponse.json({ error: 'invalid category' }, { status: 400 })
  }

  // 4. Create post in Supabase and return the URL
  const { post_id, slug } = await createPost(body, 'published')
  return NextResponse.json({ success: true, post_id, slug,
    url: `https://aitrends.ng/post/${slug}`
  })
}

The Scout Agent sends the API key in a request header:

publisher.js (scout-agent)
const response = await fetch(process.env.BLOG_API_URL, {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json',
    'x-api-key': process.env.SCOUT_API_KEY  // ← the shared secret
  },
  body: JSON.stringify({ title, content, excerpt, category, ... })
})
⚠️
SCOUT_API_KEY must be in sync in three places

The key exists in: (1) scout-agent/.env locally, (2) GitHub Actions Secrets, and (3) Vercel environment variables. If any one of them is different, Scout's publishes will silently return 401 and nothing will be posted.

Handling AI-Generated Images — What NOT to Do

A key lesson: never use Next.js <Image> for Pollinations URLs. The Next.js image optimizer fetches external images server-side on Vercel with a ~10 second timeout. Pollinations generates images on first request (15–60 seconds). Result: timeout, broken image.

The correct approach for Pollinations/external AI images — plain <img> with a fallback:

components/PostCard.tsx
// ✅ Correct: plain img tag, browser waits, gradient shows if it fails
<div style={{ background: 'linear-gradient(135deg, #0f172a, #1e3a5f)',
               position: 'relative', aspectRatio: '16/9', overflow: 'hidden' }}>
  {post.cover_image_url && (
    <img
      src={post.cover_image_url}
      alt={post.title}
      loading="lazy"
      style={{ position: 'absolute', inset: 0,
               width: '100%', height: '100%', objectFit: 'cover' }}
      onError={(e) => {
        // Hide img silently — gradient placeholder shows through
        (e.currentTarget as HTMLImageElement).style.display = 'none'
      }}
    />
  )}
</div>

The Scout Agent — Autonomous Publishing

The 13-Step Pipeline (Phase 1 Only)

Scout's Phase 1 runs every 6 hours. Its job is to queue work — not to publish. Publishing is Phase 2.

The Phase 1 pipeline
Step  0: Read #scout-editor Slack channel (Felix's editorial inputs, last 8h)
Step  1: Load evergreen vocabulary from Supabase (40 baseline + learned terms)
Step  2: Fetch all 25 RSS feeds (Africa-specific + global)
Step  3: Deduplicate — skip articles already in scout_memory
Step  4: Score each article by evergreen potential (vocab match count)
Step  5: Group by category, sort by score, cap at MAX_ARTICLES = 5
Step  6: Skip categories with fewer than MIN_ARTICLES = 2 fresh articles
Step  7: Generate digest with Gemini 3.5 Flash (Africa-first, 800+ words)
Step  8: Validate Gemini output (title + content ≥ 400 chars)
Step  9: Submit image generation job to active provider (HF/Fal/AI Horde)
Step 10: Save to pending_posts table (status: pending_image)
Step 11: Save trending terms to evergreen_vocab in Supabase
Step 12: Mark articles as seen in scout_memory (prevents re-processing)
Step 13: EXIT. Nothing is published here.

Evergreen Scoring Algorithm

Before sending articles to Gemini, Scout scores each one for long-term search value. Articles mentioning evergreen entities (major AI companies, models, concepts) are ranked higher. This ensures the highest-value stories always get into each digest. 

scout.js — evergreenScore()
const EVERGREEN_ENTITIES = [
  'anthropic', 'claude', 'openai', 'chatgpt', 'gpt-5',
  'google deepmind', 'gemini', 'microsoft', 'llama',
  'llm', 'large language model', 'ai agent', 'agentic',
  'africa', 'nigeria', 'lagos', 'techcabal', 'flutterwave'
  // ... 40+ terms
]

function evergreenScore(article, vocab = EVERGREEN_ENTITIES) {
  const text = `${article.title}  ${article.summary}`.toLowerCase()
  return vocab.filter(term => text.includes(term)).length
}

// Sort articles highest score first, then cap at 5
const sorted = [...articles].sort((a, b) =>
  evergreenScore(b, allVocab) - evergreenScore(a, allVocab)
)
const capped = sorted.slice(0, MAX_ARTICLES)

Self-Updating Vocabulary

After each successful digest, Gemini returns a list of TRENDING_TERMS — proper nouns from the story that people will search for in two years. These are saved to a evergreen_vocab Supabase table and merged with the baseline on the next run. The vocabulary grows itself. 

memory.js — saveEvergreens()
export async function saveEvergreens(terms) {
  const rows = terms
    .map(t => t.trim().toLowerCase())
    .filter(t => t.length > 2)
    .map(term => ({ term, source: 'auto' }))

  await supabase
    .from('evergreen_vocab')
    .upsert(rows, { onConflict: 'term', ignoreDuplicates: true })
}

Africa-First Gemini Prompt Engineering

The Gemini prompt is the heart of the editorial quality. The key upgrade from a generic digest to an Africa-first publication was replacing a single closing sentence with a full editorial mandate.

The old approach (wrong):

// ❌ Africa as afterthought — one sentence at end of each section
"End each story section with one sentence on the specific
implication for African developers and builders."

The new approach (correct):

// ✅ Africa-first mandate woven through the entire prompt
AFRICA-FIRST MANDATE — THIS IS NON-NEGOTIABLE:
- Every post must be written FROM an African perspective
- If source articles are about global AI events, translate them
  into African context: What does this mean for a developer in
  Lagos building on this API? What is the opportunity or threat
  for a startup in Accra?
- Lead paragraphs must establish Africa/Nigeria/West Africa
  relevance FIRST — not as an afterthought
- Close with: <p><strong>Bottom line for African builders:
  </strong> one sharp Africa-specific takeaway</p>
⚠️
Content without Africa source material stays generic

No amount of prompt engineering fixes the underlying problem that all 10 original RSS feeds were US/global sources. Adding TechCabal, Techpoint Africa, Disrupt Africa, and Ventureburn was the only real fix. You cannot prompt your way to Africa-first content if the source material has no Africa in it.

The 4-Section Post Structure

Every post generated by Gemini must follow exactly four sections in this order. This is Felix's editorial specification — do not change the order or add sections.

SectionPurposeRule
<h3>Why it matters</h3>Analysis first — impact for Nigeria/AfricaThis comes BEFORE the facts. Explain what changes for Nigerian businesses, students, or creators.
<h3>What happened</h3>The facts — who, what, where, whenName the specific person or organisation. Include a direct quote if available; paraphrase with attribution if not.
<h3>The bigger picture</h3>Context and ecosystem implicationsInfrastructure, regulation, opportunity, risk for Africa. Reference precedents where available.
<h3>What's next</h3>Forward-looking conclusionWhat to watch, what to expect. Closes with <strong>Bottom line for African builders:</strong>
💡
SEO rule: h3 headings must be keyword-specific, not generic labels

The four sections are thematic guides — they tell Gemini WHAT content goes in each section. But the h3 heading text itself must contain the primary keyword of the specific story.

✅ Correct: <h3>Why Nigeria's AI Deepfake Threat Demands Immediate Action</h3>

❌ Wrong: <h3>Why it matters</h3> — generic, not searchable, SEO-neutral

Google ranks h3 headings as signals for topical relevance. Generic headings waste that signal.

⚠️
Do not add sections. Do not reorder. Always include People Also Ask.

All four h3 sections + the People Also Ask block must address the SAME story throughout. If Gemini returns posts with more than four h3 sections (plus PAA), the structure prompt needs tightening — not new examples.

Word count: 800 words minimum — hard SEO floor. This was originally set for SEO and must not be reduced. Posts under 800 words are too thin for Google to rank competitively in the AI news vertical.

People Also Ask — What It Actually Is on AITrends.ng

"People Also Ask" on AITrends.ng is a frontend section showing hyperlinked titles of other published posts from the same category. It is NOT a Q&A block generated by Gemini inside the article body.

  • It lives in app/post/[slug]/page.tsx — outside the article body
  • It does NOT count toward the 800-word editorial minimum
  • It reuses the existing getRelated() Supabase query (same category, 3 posts, newest first)
  • Each item shows the post title as a clickable link + the excerpt below it
  • Navigates to /post/{slug} on click

SEO value: Internal linking between posts helps Google discover and index all content, distributes PageRank across the site, and reduces bounce rate — all recognised ranking signals.

Never add Q&A pairs to the Gemini prompt and call them "People Also Ask". The Q&A was removed from gemini.js (commit 21f84bc) when this distinction was clarified.

Deduplication with scout_memory

Without deduplication, Scout would re-publish the same articles every 6 hours. The scout_memory table records every article processed, keyed by the feed URL and the article's unique GUID. 

memory.js — hasSeen() and markSeen()
// Called before processing an article — throws on DB error (intentional)
export async function hasSeen(feedUrl, itemGuid) {
  const { data, error } = await supabase
    .from('scout_memory')
    .select('id')
    .eq('feed_url', feedUrl)
    .eq('item_guid', itemGuid)
    .maybeSingle()

  if (error) throw new Error(`hasSeen failed: ${error.message}`)
  return data !== null
}

// Called after saving to pending_posts — retries once on failure
export async function markSeen(feedUrl, guid, title, postId = null) {
  const row = { feed_url: feedUrl, item_guid: guid, item_title: title, post_id: postId }

  for (let attempt = 1; attempt <= 2; attempt++) {
    const { error } = await supabase.from('scout_memory').insert(row)
    if (!error) return
    if (attempt < 2) {
      console.error(`markSeen attempt ${attempt}: retrying...`)
      await new Promise(r => setTimeout(r, 2000))
    }
  }
}
🔴
hasSeen() must throw — never silently return false

If Supabase is down and hasSeen() returns false (treating all articles as new), Scout will re-publish every article it has ever seen as duplicate posts. Throwing on error halts the run immediately — a much safer failure mode.

GitHub Actions — Automated Scheduling

Phase 1: Scout Workflow (scout.yml)

.github/workflows/scout.yml
name: Scout Agent

on:
  schedule:
    - cron: '0 */6 * * *'    # every 6 hours: 00:00, 06:00, 12:00, 18:00 UTC
  workflow_dispatch:        # also allow manual trigger from GitHub UI

jobs:
  run-scout:
    runs-on: ubuntu-latest
    timeout-minutes: 15
    env:
      FORCE_JAVASCRIPT_ACTIONS_TO_NODE24: true  # silence Node 20 warnings

    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '24'
      - run: npm install

      - name: Run Scout (Phase 1 — queue posts)
        env:
          SUPABASE_URL:        ${{ secrets.SUPABASE_URL }}
          SUPABASE_SECRET_KEY: ${{ secrets.SUPABASE_SECRET_KEY }}
          GEMINI_API_KEY:      ${{ secrets.GEMINI_API_KEY }}
          IMAGE_PROVIDER:      ${{ secrets.IMAGE_PROVIDER }}
          HF_API_TOKEN:        ${{ secrets.HF_API_TOKEN }}
          SLACK_BOT_TOKEN:     ${{ secrets.SLACK_BOT_TOKEN }}
          # ... more secrets
        run: |
          node -e "import('./scout.js').then(m => m.runScout())
            .then(() => process.exit(0))
            .catch(e => { console.error(e); process.exit(1) })"

      - name: Notify Slack on failure
        if: failure()
        run: |
          curl -s -X POST "$SLACK_WEBHOOK_URL"             -H 'Content-Type: application/json'             -d '{"text":"❌ Scout Agent failed"}'
⚠️
process.exit(0) is mandatory

Supabase realtime keeps a WebSocket open after all queries complete. Without process.exit(0), the Node.js process hangs indefinitely until the GHA timeout kills it. This was discovered when a run took 502 seconds instead of 78 — the 430-second gap was the WebSocket hanging.

GHA Cron Reliability — Two External Triggers + One Fallback

GitHub Actions scheduled crons are not reliable at short intervals. This was discovered the hard way: */5 * * * * ran only 2 times in 13 hours. */15 * * * * ran 0 times after deployment. Posts sat in the pending queue for 6+ hours waiting for Phase 2 to fire.

⚠️
GHA cron is best-effort, not guaranteed

GitHub does not guarantee scheduled workflows fire at the exact time, or at all. Under load, GHA deprioritises or skips short-interval schedules entirely. Never design a production pipeline that depends on GHA cron firing reliably at intervals shorter than 1 hour.

The fix is two independent external trigger layers, plus GHA as an unreliable fallback:

The Stuck-Post Guard

A subtle but critical safety mechanism: if a GHA runner is killed mid-Phase-2 (timeout, network failure, etc.), a post can get stuck in 'generating' status forever. The guard detects and recovers these posts automatically. 

memory.js — getPendingPosts() with stuck-guard
export async function getPendingPosts() {
  // Reset posts stuck in 'generating' for more than 15 minutes
  // (handles GHA runner crashes mid-process)
  const stuckCutoff = new Date(Date.now() - 15 * 60 * 1000).toISOString()
  await supabase
    .from('pending_posts')
    .update({ status: 'pending_image', updated_at: new Date().toISOString() })
    .eq('status', 'generating')
    .lt('updated_at', stuckCutoff)

  // Now fetch all posts ready for processing
  const { data, error } = await supabase
    .from('pending_posts')
    .select('*')
    .eq('status', 'pending_image')
    .order('created_at', { ascending: true })

  if (error) throw new Error(`getPendingPosts failed: ${error.message}`)
  return data || []
}

V3: Source Article Photograph (Primary)

Added in Session #5 (6 June 2026) — this is now the primary image strategy.

Every news article has an editorial photograph. When a Nigerian newspaper writes about the CBN Governor announcing an interest rate cut, they publish a photograph of the governor at the podium. That photograph is the correct image for the story. No AI generator can produce it.

The new image priority order:

  1. Source article og:image — fetch the real photograph from the source article URL
  2. AI illustration (paint/pencil only) — fall back to HuggingFace FLUX if no source image

Implementation in complete.js — trySourceImage(sourceUrls): 

// Fetches the og:image meta tag from the source article
// Returns { done, imageBuffer, contentType } or null
async function trySourceImage(sourceUrls) {
  const res = await fetch(sourceUrls[0], { signal: AbortSignal.timeout(8000) })
  const html = await res.text()
  const match = html.match(/<meta[^>]+property=["']og:image["'][^>]+content=["']([^"']+)["']/i)
  const imgRes = await fetch(match[1])
  const imageBuffer = Buffer.from(await imgRes.arrayBuffer())
  if (imageBuffer.length < 5000) return null  // reject icons/placeholders
  return { done: true, imageBuffer, contentType }
}

Why og:image and not scraping the first <img> tag

The og:image Open Graph tag is specifically set by editors for sharing — it is the image the publication CHOSE to represent the article. Nigerian newspapers set this to the editorial photo of the key person or event. It is more reliable than the first image in the HTML (which is often a logo or ad).

  • Watercolour or oil paint — for hopeful, positive, energetic stories
  • Pencil sketch — for serious, analytical, cautionary stories

Risograph, vector illustration, and editorial ink sketch were removed. They produced abstract art that did not visually communicate the story.

V4: The Fabrication Fix — Why RSS Summaries Are Not Enough

When Gemini was only receiving RSS summaries (100-300 characters) per article, then being asked to write 800+ words with named people, direct quotes, and specific African facts, it fabricated content to fill the gap. This is not misbehaviour — it is a predictable response to impossible constraints given insufficient source material.

⚠️
Never ask an LLM to write detailed factual content from summaries

A 150-character RSS summary can produce a 10-word accurate sentence. Ask for 800 words from that same summary and the model will fill the remaining 790 words with plausible-sounding invented content. The fabrication is proportional to the gap between source material and required output length.

Debugging War Stories

The most valuable lessons come from things that break. Here are the seven most instructive debugging episodes from this project.

War Story 1: The Node.js 18 Crash That Hid Itself

For the first week of the Scout Agent, every GHA run showed green ✅. But no posts were being published. The pipeline appeared to work perfectly.

The reveal: GitHub was running the workflow on Node.js 18, which is deprecated by Supabase. The @supabase/supabase-js package throws immediately on startup if it detects Node 18 — before Scout even fetches a single RSS article. The original workflow used || true to suppress all errors, so even a total crash showed as green. 

// The broken workflow — silent failure pattern
run: |
  timeout 110 node index.js || true  ← always exits 0, hides all errors

// Proof of the crash: every "successful" run took exactly 16 seconds
// 16 seconds = startup time + crash. Real runs take 78–120 seconds.

The fix was two changes:

  1. Upgrade to node-version: '24' in the workflow
  2. Remove || true — let the workflow fail loudly when Scout fails
  3. Add a Slack failure alert so you know immediately when runs fail

War Story 2: The Supabase WebSocket That Hung Everything

After the Node 18 fix, Scout ran successfully — but GHA runs took 502 seconds. The run log showed posts were published after ~70 seconds, then... nothing. 430 seconds of silence before the timeout killed the process.

Root cause: Supabase's JavaScript client opens a realtime WebSocket to listen for database changes. It keeps this socket alive indefinitely. When runScout() finishes, Node.js doesn't exit because the WebSocket is still open. 

// Before: process hangs for 430 seconds until GHA timeout
run: node -e "import('./scout.js').then(m => m.runScout())"

// After: explicitly exit when done — WebSocket is destroyed
run: node -e "import('./scout.js')
  .then(m => m.runScout())
  .then(() => process.exit(0))      ← this is the fix
  .catch(e => { console.error(e); process.exit(1) })"

// Result: runs went from 502 seconds to 78 seconds

Images stopped loading on the site. The browser DevTools showed all Pollinations image requests returning HTTP 402. Assumed to be rate limiting.

The migration script was run from GitHub Actions (a fresh IP, not rate-limited). Result: 26 posts, 26 failures. Not a rate limit — same 402 from a clean IP.

War Story 5: The Phase 2 That Never Fired

Phase 1 ran on schedule at 11:50 UTC. It found 18 fresh articles, picked the best one, generated a digest, and saved it to pending_posts. Then it exited cleanly. The Slack notification said: "Phase 2 will publish when images are ready." Six hours later — nothing published.

War Story 6: HTTP 403 — The Token That Couldn't Trigger Its Own Sibling

After adding the Phase 1 → Phase 2 direct trigger to scout.yml, the next Phase 1 run immediately failed. The run log showed Phase 1 had run perfectly — found articles, queued a post — but crashed on the last step: 

Triggering Phase 2 (complete.yml)...
HTTP 403: Resource not accessible by integration
(https://api.github.com/repos/FAIT-Blog/scout-agent/actions/workflows/complete.yml)
Process completed with exit code 1.

The command was gh workflow run complete.yml --repo FAIT-Blog/scout-agent, using GH_TOKEN: ${{ secrets.GITHUB_TOKEN }}. The token was valid. The repo existed. The workflow existed. And yet: 403.

Root cause: GITHUB_TOKEN in GHA workflows has read-only access to Actions by default. To dispatch another workflow, the workflow must explicitly request write access with a permissions block. Without it, any call to the Actions API returns 403. 

scout.yml — the one-line fix
# Add this ABOVE the jobs: block
permissions:
  actions: write    # allows GITHUB_TOKEN to dispatch workflow_dispatch events

jobs:
  run-scout:
    # ...

Lesson: GITHUB_TOKEN permissions are additive — you must explicitly grant what you need. The default token can read code and write to the repo, but cannot control Actions. Always add permissions: actions: write to any workflow that needs to trigger another workflow. This is a one-line fix but takes real debugging to find.

⚠️
Part 2 — The fix worked for manual runs only

Adding permissions: actions: write fixed the 403 when Phase 1 was triggered manually via workflow_dispatch. But when the GHA cron fired Phase 1 on its regular schedule, the same HTTP 403 came back. The permission block was in the YAML. The token was the same. Same repo, same workflow. Still 403.

The deeper restriction: GITHUB_TOKEN on schedule-triggered runs cannot dispatch workflow_dispatch events, even with actions: write. GitHub prevents this specifically to stop scheduled workflows from creating automated dispatch loops. The fix only applies to workflow_dispatch-triggered parent runs.

The real solution: remove the step entirely and let cron-job.org handle Phase 2 triggering via an external PAT, which has no such restriction. See War Story 8.

War Story 7: The PAT That Got a 404 on a Repo That Exists

Setting up cron-job.org to trigger Phase 2 externally. Created a GitHub fine-grained Personal Access Token (PAT), pasted it into cron-job.org, clicked "Run now". Result: 404 Not Found.

The repo definitely exists. The URL was correct. Direct curl test: 

curl -s -o /dev/null -w "%{http_code}"   -X POST   -H "Authorization: Bearer github_pat_..."   -H "Content-Type: application/json"   -d '{"ref":"main"}'   "https://api.github.com/repos/FAIT-Blog/scout-agent/actions/workflows/complete.yml/dispatches"

→ 401

404 from cron-job.org, 401 from curl. Different codes for the same bad token — different clients handle GitHub's error responses differently.

Root cause: The PAT was created in the GitHub personal account settings (felixokon), with FAIT-Blog/scout-agent selected as the repository. But for an organisation repo, the fine-grained PAT Resource Owner must be set to the organisation (FAIT-Blog), not the personal account.

A personal-account token sees only repos owned by felixokon. When it tries to reach FAIT-Blog/scout-agent, GitHub responds as if the repo doesn't exist — hence 404/401.

Permissions → Actions: Read and write

  • Phase 1 was marking itself as "failed" on every 6-hour cron run

  • The content pipeline continued working (Phase 2 published posts via cron-job.org) but Phase 1 appeared broken
  • Every Slack "Scout Agent failed" alert was a false alarm — content was actually queued

A recurring failure pattern in this project: a problem is identified, Claude Code makes a change, tests it, and announces the fix. The Editorial Architecture

This is the part that took the longest to get right. Getting the automation to run reliably was the engineering problem. Getting it to write good journalism was the editorial problem. They are completely different problems. The second one is harder.

The Africa Relevance GATE

For months, posts were being published with fabricated Africa connections. "African developers face expensive cloud infrastructure costs and this UK developer's micropython-wasm library could help them." That is not journalism. That is a language model obeying instructions it cannot fulfil with the source material available.

The fix was to give Gemini an explicit rejection gate. The first thing the prompt says now is: 

AFRICA RELEVANCE GATE — CHECK THIS BEFORE WRITING ANYTHING:

GENUINE relevance means at least ONE of these is true:
- A Nigerian, Ghanaian, Kenyan, or other African organisation is directly involved
- The event happened IN Africa or is directly targeting Africa's market
- Multiple African news outlets are covering this very story
- An Africa-based company raised funding or launched a product

FABRICATED relevance (DO NOT accept these):
- "African developers could use this tool"
- "This US technology could help Nigerian startups"
- Any connection that applies equally to any country in the world

IF the articles DO NOT meet the genuine relevance test, output EXACTLY:
SKIP_NO_AFRICA_RELEVANCE

When Gemini outputs that signal, scout.js catches it, marks the articles as seen (so they do not re-queue), and moves on silently. No post is published. No error is thrown. It just quietly skips a category the way a good editor would skip a story that does not belong in the publication.

The SOURCE RULES — How Fabrication Was Stopped

Discovered in Session #5: Gemini was receiving ~150 characters of RSS summary per article, then being asked to write 800 words with named people, quotes, and African-specific facts. Of course it invented. There was nothing else to work with.

The fix was two-pronged. First, fetcher.js was connected to the main pipeline — fetchContent(article.link) now runs for every article before Gemini sees it. This gives Gemini up to 4,000 characters of real article body to work from. Second, explicit SOURCE RULES were added to the prompt: 

SOURCE RULES — strictly enforced:
- Write ONLY from content provided in the FULL TEXT fields above.
- Never fabricate quotes. If no verbatim quote appears in FULL TEXT,
  attribute clearly: "according to [Source name]" and paraphrase.
- If the source material does not contain enough detail to support
  a section, write what is known — do not pad with invented context.
- If a claim cannot be traced to the FULL TEXT provided, do not include it.

Gemini's job changed from "write an 800-word editorial about AI in Africa" to "summarise, analyse, and contextualise the specific articles I am giving you." That is a much more honest task, and Gemini performs it much more honestly.

import { Jimp, JimpMime } from 'jimp'

async function styleWithAI(imageBuffer, imagePrompt) {
  const isPencil = /pencil sketch|serious|analytical|cautionary/i.test(imagePrompt)
  const img = await Jimp.fromBuffer(imageBuffer)

  img.cover({ w: 1200, h: 630 })  // crop to cover dimensions

  if (isPencil) {
    img.color([{ apply: 'desaturate', params: [55] }])  // stark analytical feel
    img.contrast(0.25)
    img.brightness(-0.08)
  } else {
    img.color([{ apply: 'desaturate', params: [20] }])  // painterly warmth
    img.contrast(0.12)
    img.brightness(-0.04)
  }

  const overlay = new Jimp({ width: 1200, height: 630, color: 0x0a0a2a47 })
  img.composite(overlay, 0, 0)

  return { done: true,
    imageBuffer: await img.getBuffer(JimpMime.jpeg, { quality: 88 }),
    contentType: 'image/jpeg'

War Story 11: The Black Images

The Phase 2 logs said: "📸 Source photo found (603KB) — recreating in art style... 🎨 Source photo recreated as AI illustration (21KB)". This was treated as confirmation that jimp was working. It was not.

The 21KB output was suspicious — a 1200×630 JPEG with real content should be 80–200KB. A pixel sample revealed the truth: 

// Sample pixels from cover_62.jpg (jimp-styled)
(0,0)     → rgb(1, 2, 10)
(600,315) → rgb(1, 2, 10)   ← centre
(1199,629) → rgb(1, 2, 10)

// Every single pixel the same near-black colour — rgb(1,2,10) ≈ #01020a

Root cause:

When jimp loads a JPEG, JPEG images have no alpha channel. jimp treated every pixel of the loaded photo as having alpha = 0 (fully transparent). In the composite operation (BLEND_SOURCE_OVER): 

output = overlay × overlay_alpha + background × background_alpha × (1 - overlay_alpha)
       = overlay × 0.278 + background × 0 × 0.722
       = rgb(10,10,42) × 0.278
       = rgb(2.8, 2.8, 11.7)  ≈  rgb(1, 2, 10) ✗

The dark blue overlay at 28% opacity, with the background effectively transparent, produced a near-black image on every run. All 7 jimp-styled covers were this colour.

🔴
Log output is not proof of correctness

The Phase 2 log said "🎨 Source photo recreated as AI illustration (21KB)" — which sounds like success. The only actual proof was downloading the image and sampling pixels. A small file size (21KB for a 1200×630 JPEG) is a strong signal that something is wrong. Always verify image quality visually or by pixel inspection when implementing image processing, not just by log output.

The Problem: General News Infiltrating the AI Pipeline

After the Session #8 simplification (Gemini accurate-rewrite prompt + single article), a 5-post audit revealed that the content quality had improved significantly — but a new category of failure appeared: posts about topics that had nothing to do with AI at all.

Root Cause: The Africa GATE Does Not Check for AI

The pipeline had two filters at this point:

  1. Evergreen scoring — ranks articles by how many of 40+ AI/tech vocabulary terms they contain. This favours AI articles but does not exclude non-AI articles — a fintech article mentioning "startups", "funding", or "Nigeria" can score high.
  2. Africa GATE (Gemini call) — asks Gemini: does this story have a genuine African/Nigerian angle? If no: SKIP_NO_AFRICA_RELEVANCE. This correctly blocks global tech news with no Africa connection — but the GATE only checks geography, not topic domain.

The gap: an article about Nigerian football teams using spreadsheets could pass both filters. It is Nigerian (passes Africa GATE), and might mention "strategy" or "data" (gets an evergreen score). Nothing in the pipeline asked the question: is this about artificial intelligence?

The Fix: hasAISignal() — Implicit Filter

An explicit AI keyword check was added to scout.js as an early filter — before evergreen scoring, before the Gemini call: 

scout.js — hasAISignal() (commit 000c25f)
const AI_SIGNAL_TERMS = [
  'ai', 'artificial intelligence', 'machine learning', 'deep learning',
  'neural', 'llm', 'large language model', 'generative', 'chatbot',
  'automation', 'algorithm', 'nlp', 'computer vision', 'robotics',
  'openai', 'anthropic', 'google ai', 'gemini', 'claude', 'gpt',
  'huggingface', 'mistral', 'model', 'training data', 'inference'
];

function hasAISignal(article) {
  const text = ((article.title || '') + ' ' + (article.fullText || '')).toLowerCase();
  return AI_SIGNAL_TERMS.some(term => text.includes(term));
}

// Applied in runScout() before evergreenScore():
const aiFiltered = deduped.filter(a => hasAISignal(a));
console.log(`🤖 AI signal filter: ${deduped.length} → ${aiFiltered.length} articles`);

Every article is checked against the list. If neither the title nor the fetched body contains a single AI term, the article is dropped immediately — no Gemini call, no evergreen scoring, no cost.

Chapter 17 — Redesigning the Editorial Channel: From Polling to Push

Session #9 · 12 June 2026 · scout-agent + aitrends.ng

The Original Design and Its Problem

From the beginning, Scout included an editorial channel: Felix could post messages to a private Slack channel (#scout-editor) and Scout would pick them up at the start of each run. The implementation used conversations.list to find the channel ID and conversations.history to fetch the last 8 hours of messages.

This worked — but it was architecturally wrong. Scout was treating Slack as a database. Every 6-hour run made 2 Slack API calls just to read a few messages that might not even exist. If the lookback window was too narrow, messages posted near a boundary could be missed or double-processed. There was no acknowledgement — Felix had no way to know if his input had been used. And if the Slack bot's token was revoked or the API call failed, the pipeline would crash at step zero.

When Felix asked "can you help me look at what can be optimised?", the instinct was to patch around the edges: add caching, tune the lookback window, handle errors better. But the better question was: what does this system actually need to be?

The New Design: Events Over Polling

The redesign flips the direction of data flow. Instead of Scout pulling from Slack, Slack pushes to a webhook endpoint the moment Felix posts a message.

Felix posts to #scout-editor ↓ Slack Events API ↓ POST /api/slack/editorial (aitrends.ng — always running on Vercel) ↓ editorial_queue table (Supabase) ↓ felix.js reads pending rows (at Scout run time) ↓ consumed after a post is queued ↓ complete.js replies in Slack thread: "✅ Used in: [title]"

Slack is now a write path, not a storage system. The DB is the single source of truth. Felix's inputs survive Slack's 90-day history limit. Scout's startup is not gated on a Slack API call. And for the first time, Felix gets feedback: a threaded reply in his original message confirming which post his input contributed to.

File-by-File Walkthrough

1. Supabase table: editorial_queue

Seven columns that matter:

  • slack_ts — the Slack message timestamp. Unique. Used as thread_ts when replying.
  • slack_channel — the channel ID. Needed to target the reply.
  • urls — text[]. URLs extracted from the message.
  • notes — any non-URL text annotation Felix included.
  • status — pending | consumed. Scout only reads pending rows.
  • used_in_post — post title written here when consumed. Matched by complete.js for the reply.
  • posted_at — NULL until the Slack reply has been sent. Prevents double-replies.

The UNIQUE constraint on slack_ts makes the insert idempotent: if Slack retries the webhook because our response was slow, the second insert silently fails with a 23505 duplicate-key error, which the endpoint catches and ignores.

2. The webhook endpoint: /api/slack/editorial

Lives in aitrends.ng (Vercel — always running). Three jobs:

  1. url_verification — when you first point Slack at a request URL, Slack sends a JSON body with "type": "url_verification" and a challenge string. The endpoint must echo it back. Without this, Slack refuses to save the URL.
  2. Signature validation — every other request is validated with HMAC-SHA256. Slack signs requests using your app's Signing Secret: v0={hex(sha256(SIGNING_SECRET, "v0:{timestamp}:{raw_body}"))}. Requests older than 5 minutes are rejected (replay attack guard). This is what SLACK_SIGNING_SECRET is for.
  3. Message handling — extract URLs and plain text from the message, insert into editorial_queue, return 200 OK within 3 seconds. (Slack will retry if you take longer.) 
// Slack sends this on first-time setup — must respond with the challenge
if (body.type === 'url_verification') {
  return NextResponse.json({ challenge: body.challenge })
}

// Validate signature on all real requests
if (!verifySlackSignature(signingSecret, signature, timestamp, rawBody)) {
  return NextResponse.json({ error: 'Invalid signature' }, { status: 401 })
}

// Ignore retries we've already processed
if (req.headers.get('x-slack-retry-num')) {
  return NextResponse.json({ ok: true })
}

3. felix.js — 126 lines → 60 lines

The getChannelId() function, the pagination loop, the 8-hour lookback calculation, the conversations.history call — all gone. Replaced with one Supabase query. The return value gains a rowIds field alongside topics and sources, so scout.js knows which rows to mark consumed after a successful queue.

URL content is still fetched here, at Scout run time — not at webhook time. This keeps the webhook response under 3 seconds and avoids fetching content for inputs that ultimately don't get used.

4. memory.js — four new functions

  • getEditorialQueue() — SELECT pending rows, ordered oldest-first.
  • consumeEditorialRows(ids, postTitle) — UPDATE where status='pending'. The WHERE status='pending' guard makes this idempotent: if two categories both queue from the same felix inputs in one run, the second call is a no-op.
  • getPendingEditorialReplies(postTitle) — SELECT consumed rows matched by post title where posted_at IS NULL.
  • markEditorialReplied(ids) — SET posted_at = now(). Prevents double-replies if Phase 2 cycles again.

5. slack.js — replyToFelix()

Uses chat.postMessage with thread_ts set to the original message's Slack timestamp. The reply appears indented under Felix's original message — the standard Slack threading model. Message text: "✅ Used in: [post title]\n[live URL]"

6. complete.js — closing the loop

After notifySlack() completes, Phase 2 now does: 

const replyRows = await getPendingEditorialReplies(post.title)
if (replyRows.length > 0) {
  for (const row of replyRows) {
    await replyToFelix(row.slack_ts, row.slack_channel, post.title, postUrl)
  }
  await markEditorialReplied(replyRows.map(r => r.id))
}

The match is by post title, not by ID. This works because consumeEditorialRows writes the title into used_in_post in Phase 1, and Phase 2 has the same title from pending_posts.title. The title is the join key.

The Vibe-Coding Lesson: Patch vs Redesign

When Felix asked about optimisations, Claude's first instinct was a list of patches: cache the channel ID, handle the window boundary, add better error handling. These would have all improved the code without fixing the core problem.

The better move was to step back and ask: what is this code trying to do? It tries to get Felix's inputs into the pipeline. The Slack API is an implementation detail — a bad one. Once the real goal was named, the right architecture was obvious: a webhook that writes to a DB, and a reader that reads from a DB. Slack plays to its strengths (real-time delivery). Supabase plays to its strengths (persistent, queryable state). Scout plays to its strengths (runs when it's ready, reads what it needs).

"Don't just build on this one. What would you do differently instead of patching this one?" — Felix, Session #9

This is the right instinct. When a module feels like it's fighting the architecture — when you're adding retries and caches and boundary conditions — ask whether the module is in the wrong position entirely. Sometimes the best refactor is to invert the data flow direction.

Immediate Dispatch — No More 6-Hour Wait

After the editorial system was built, Felix asked the right follow-up question: why should an editorial input wait up to 6 hours for the next Phase 1 cron tick? The whole point of the system is to act on a signal NOW.

The fix was one addition to the webhook endpoint: after saving the DB row, fire a workflow_dispatch to scout.yml via the GitHub Actions API. Phase 1 runs within seconds. The dispatch is non-blocking — the 200 OK is returned to Slack immediately, and the GitHub API call runs in the background. A dispatch failure is logged but never breaks the webhook; the row is already safely in Supabase and the scheduled 6-hour run will pick it up at worst.

This requires one env var in Vercel: GITHUB_PAT — a fine-grained personal access token with Actions: read and write scope on FAIT-Blog/scout-agent. The existing aitrends-cron-trigger token already had exactly these permissions, so it was regenerated and reused.

The Editorial Override — Exempting Felix From the Africa GATE

Session #10 · 13 June 2026 · scout-agent commit 221154e

The Problem

The Africa GATE was doing its job: Gemini returned SKIP_NO_AFRICA_RELEVANCE for any category whose RSS articles had no genuine African angle. But this was also blocking Felix's own editorial submissions. When Felix posted a URL to #scout-editor — for example, a Substack about Claude 4 capabilities — Scout passed it as context to Gemini alongside the best RSS article. If that RSS article had no Africa relevance, Gemini SKIP'd the entire category. Felix's URL was ignored.

Felix's instruction: "What if the Africa and Nigeria GATE is configured to exempt my recommendations? I just want to see if and how the process works so we can fine-tune it perfectly."

The distinction is important. The Africa GATE exists to stop the system from fabricating Africa connections to global news stories. But when Felix submits a URL, the editorial decision has already been made by a human. The gate should not override the editor. The editor is the gate.

The Solution: Step 0.5 — Editorial Override

In scout.js, a new block was added between STEP 0 (load editorial inputs) and STEP 1 (fetch RSS feeds). When felixInputs.sources.length > 0 — meaning Felix submitted a URL whose content was successfully extracted — the pipeline generates a dedicated post from that content before the normal RSS loop begins.

The Africa GATE is suspended for this call. A new Gemini function, generateEditorialDigest(felixInputs), handles it. The prompt says: 

EDITOR DIRECTIVE: The editor (Felix) has personally submitted this content
for immediate coverage. This is an editorial override — the Africa GATE is
suspended for this request. Write a post about this content framed for an
African AI audience.

Gemini still writes for an African AI audience — it still frames relevance at the end — but it cannot return SKIP. Felix's direction is the editorial authority.

After the post is queued, consumeEditorialRows() marks the editorial_queue rows as consumed. The normal RSS loop then runs as usual; if a category also succeeds, an additional post may be queued. That is intentional — Felix submitted something AND the RSS had good content. Both publish.

Lesson: The Gate Serves the System — Not the Other Way Around

Quality gates exist for a specific threat. The Africa GATE exists to stop the system from inventing Africa connections. When the editor makes a deliberate choice, that threat disappears. The gate must not apply to the person whose judgment created it.

Build every automated guard with a human override path. Not a bypass that disables the guard — a path that correctly identifies "a human decided this" and acts accordingly.

Why Only URL Submissions Trigger the Override

If Felix posts a URL, felix.js extracts the article body (up to 4000 chars) and stores it in felixInputs.sources. That gives Gemini real source material to rewrite accurately. The override requires this — without actual content, Gemini would have to fabricate, which is exactly what the whole system is built to prevent.

If Felix posts only plain text (no URL), it goes into felixInputs.topics and is passed as editorial context to each category's normal Gemini call. That behaviour is unchanged. The override only fires when there is extractable source material.

The MIN_ARTICLES Change — From 2 to 1

At the same time, MIN_ARTICLES was lowered from 2 to 1. The logic: the Africa GATE is the quality guard, not article count. Requiring two AI-signal articles before even calling Gemini was the wrong lever. It caused entire runs to skip when only one strong AI article existed in the category — which happened often with dedicated African AI portals that post infrequently.

One strong, AI-relevant article is enough for Gemini to evaluate and the Africa GATE to accept or reject. The gate was always the real filter. MIN_ARTICLES=2 was redundant friction that suppressed publishing without improving quality.

AI_SIGNAL Expansion — Catching Hyphenated Forms

The AI signal gate (hasAISignal()) used ' ai ' (with surrounding spaces) to match the word "AI". This matched "AI regulation" or "how AI is transforming" but not "AI-powered" or "AI-driven" — because the hyphen breaks the space pattern.

African tech journalism commonly uses compound forms like "AI-powered fintech" or "AI-enabled agriculture". These were failing the gate and being discarded before Gemini ever saw them. The fix: add the compound forms explicitly to the term list.

New terms added:

  • Compound: ai-powered, ai-driven, ai-enabled, ai-based, ai-generated, ai-assisted
  • Domain terms: chatbot, natural language, computer vision, predictive analytics, speech recognition, language model, multimodal, robotics

These are all areas regularly covered in African fintech and govtech reporting. Adding them improves recall without meaningfully increasing false positives — these terms rarely appear in sports, crime, or politics stories.

The Slug Truncation Fix

A cosmetic but important fix in aitrends.ng/lib/slugify.ts. The old code used .slice(0, 80) to cap slug length. This could cut a word mid-string, leaving a trailing hyphen: ...ethics-charter-featuring-.

The fix: after slicing at 80, find the last hyphen in the result. If it is in the second half of the string (past position 40), trim back to it. Then strip any remaining trailing hyphens. Slugs now always end on a complete word. URLs work either way — this was a cosmetic issue — but clean URLs matter for professionalism and SEO trust signals. 

if (slug.length > 80) {
  slug = slug.slice(0, 80)
  const lastHyphen = slug.lastIndexOf('-')
  if (lastHyphen > 40) slug = slug.slice(0, lastHyphen)
}
return slug.replace(/-+$/, '').replace(/^-+/, '')
